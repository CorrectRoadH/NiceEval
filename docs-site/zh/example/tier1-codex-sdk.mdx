对比对象：

- **before**：[`examples/zh/origin/codex-sdk/`](https://github.com/CorrectRoadH/niceeval/tree/main/examples/zh/origin/codex-sdk) —— 一个独立的 `@openai/codex-sdk` HTTP 服务，还没接任何 eval。
- **after**：[`examples/zh/tier1/codex-sdk/`](https://github.com/CorrectRoadH/niceeval/tree/main/examples/zh/tier1/codex-sdk) —— 同一个应用接入 NiceEval 之后的样子。

`src/backend/*` 逐字节未变。`agents/codex-sdk.ts` 拉起同一个子进程、调
`POST /api/chat`，把原生 `ThreadEvent`（`thread.started` / `item.started` /
`item.completed` / `turn.completed` / `turn.failed`）映射成标准事件流；没有 HITL
（Codex SDK 不支持）。声明了 `tracing`（codex CLI 原生 otel 配置段），但省略了
`spanMapper`——内置的 codex span 归一逻辑没有从 `"niceeval/adapter"` 公开导出，
黑盒示例只能拿到已发布的公开 API，走 core 的通用 heuristic 兜底。eval 测的是真实的
"在工作目录里写文件、跑命令"，不是硬凑的天气/计算器工具。

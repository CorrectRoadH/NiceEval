Before you can assert or judge anything, the agent has to act. **Drive** is the eval-side code that sends input and receives results: `t.send()`, `t.sendFile()`, `t.newSession()`, `t.respond()`, and `t.respondAll()`.

Every drive action returns or advances a **Turn**. NiceEval assertions and judges read from Turn data.

## `t.send`

```ts
const turn = await t.send("What's the weather like in Brooklyn today?");
turn.expectOk();
t.check(turn.message, includes("sunny"));
```

`t.send()` calls the selected adapter's `send(input, ctx)` once and appends the returned events to the current session line.

## `t.sendFile`

```ts
await t.sendFile("fixtures/invoice.png", "Extract the total amount.", {
  output: z.object({ amount: z.number(), currency: z.string() }),
});
```

Files arrive at the adapter as base64 `input.files`. If the turn declares `output`, the adapter returns structured `data` and the runner validates it.

## Sessions

The first turn of an eval starts a session line. Multiple sends on `t` use the same line. Use `t.newSession()` when you need an independent conversation.

```ts
await t.send("Remember that my code is 42.");
await t.send("What code did I give you?");
t.messageIncludes("42");

const other = t.newSession();
await other.send("What code did I give you?");
other.notMessageIncludes("42");
```

Session continuation is adapter work. The adapter can use `ctx.session.history()` for client-side full-history protocols or `ctx.session.id` / `ctx.session.capture()` for server-side session IDs.

## HITL responses

When a turn returns `status: "waiting"` and emits `input.requested`, the eval can answer:

```ts
const draft = await t.send("Send the email after approval.");
draft.parked();

const request = t.requireInputRequest({ action: "send_email" });
await t.respond({ request, optionId: "approve" });

t.calledTool("send_email", { status: "completed" });
```

`t.respond()` is also just another adapter `send` call on the same session line. The answer arrives in `input.responses` with a `requestId`.

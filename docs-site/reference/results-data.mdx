Underneath the building blocks in [Custom Reports](/tutorials/custom-reports) — metrics, compute functions, dual-render components — sits one more layer: `niceeval/results`, the parser for on-disk artifacts. Both official doors and every report building block read from this layer, with no private data channel; when a scoring need cannot even be expressed as a compute function, drop to this layer and grab the data directly.

When to drop to this layer:

- **The scoring cannot even be folded into a compute function.** Tables, matrices, scorecards, and scatter plots are all "folding"; distribution-style views (histograms, per-event analysis) are not folding — you have to traverse attempts yourself.
- **Feeding results into your own system.** A data warehouse, alerting, an internal platform — they consume typed data, not a disk layout.
- **Converting results from another platform into NiceEval's format.** The write side guarantees "what is written back is exactly what is read back" — once converted, `niceeval show` / `view` / the report building blocks all work on it directly.

This layer only handles reading and writing data; it holds no opinion of its own: no merging, no aggregating, no deduplicating — it faithfully reflects the disk. "How to view it" belongs to the report building blocks above it.

## Reading: `openResults`

The input is a `.niceeval/` directory (or a results root produced by `copySnapshots`, the same layout); the output is four layers of data: **experiment → snapshot (one run of an experiment) → eval → attempt**. From here on you never touch a path, never check file existence, never parse JSON:

```typescript
import { openResults } from "niceeval/results";

const results = await openResults(".niceeval");

results.experiments;           // Experiment[]: one entry per experiment, carrying its full history
results.skipped;               // Unreadable data on disk: { dir, reason, schemaVersion?, producer? }[]
```

**Layer one, experiment** — every run sharing the same experiment id is grouped together:

```typescript
const exp = results.experiments.find((e) => e.id === "compare/bub-gpt-5.4")!;

exp.snapshots;                 // Snapshot[]: historical snapshots, newest first
exp.latest;                    // the latest one (= snapshots[0])
exp.evalIds;                   // the union of evals ever covered in history, the basis for coverage detection
```

**Layer two, snapshot** — one run of an experiment. Who it is, when it ran, and **what wrote it** all live at this layer:

```typescript
const snap = exp.latest;

snap.agent;                    // this snapshot's own agent
snap.model;
snap.startedAt;                // when this run happened
snap.completedAt;              // written back at wrap-up; missing = the process was interrupted, completed attempts still read fine
snap.diagnostics;              // written back alongside completedAt: things that went wrong during this run but can't be pinned to one eval
snap.dir;                      // the snapshot directory's absolute path — the on-disk snapshot itself, there is no lower layer
snap.producer;                 // { name: "niceeval", version: "0.4.6" } — who wrote this result
snap.schemaVersion;            // the result format version
```

**Layer three, eval** — every question in this run, with attempts (retry history) hanging off the question:

```typescript
for (const ev of snap.evals) {
  ev.id;                       // "algebra/quadratic"
  ev.attempts;                 // AttemptHandle[]: every attempt for this question
}
snap.attempts;                 // when question boundaries don't matter, every attempt flattened
```

**Layer four, attempt** — slimmed-down entries are already in hand; heavy artifacts are all lazily loaded. This is the only type carrying the `Handle` suffix — the three layers above are plain data, while methods on this layer touch disk:

```typescript
const attempt = snap.evals[0].attempts[0];

attempt.evalId;                // "algebra/quadratic" — which question it belongs to, no detour through result
attempt.experimentId;          // which experiment it belongs to
attempt.result;                // EvalResult: verdict, assertions, structured error/diagnostics, usage, cost
attempt.ref;                   // { snapshot, attempt }: an evidence reference, the same identity as view deep links and report cell refs
await attempt.events();        // StreamEvent[] | null
await attempt.trace();         // TraceSpan[] | null
await attempt.o11y();          // O11ySummary | null
await attempt.diff();          // DiffData | null (can reach hundreds of MB, so it must be lazy)
await attempt.sources();       // SourceArtifact[] | null
```

Three key points:

- **Lazy loading doubles as an existence check.** A missing artifact returns `null`, never throws — when a remote agent has no `diff.json`, `await attempt.diff()` is simply `null`, and you decide from there whether to skip or report the gap.
- **Unreadable data on disk is never silent.** A version-incompatible or corrupted snapshot goes into `skipped` with a reason; whether to display it is up to you, but the gap is always computed.
- **Memoized by handle within the same process.** Reading `diff()` twice from two places never reads the same hundred-plus MB payload twice.

`attempt.result.error` is the single fatal execution error that puts an attempt into `errored`; it carries a stable `code`, a human-readable `message`, the lifecycle operation where the error occurred, and an optional bounded cause/stack. `attempt.result.diagnostics` can coexist with any verdict and holds problems found while the run could still continue, or at wrap-up. Transient `progress` is never stored on disk, and an OTel trace is not a prerequisite for storing errors.

## Oversized output is truncated

An agent runs one command and the output can be absurdly large — a recursive `grep` walks into `node_modules`, hits a minified JS file, and a single line is several MB. That output goes into both `events.json` and `trace.json`; left alone, one attempt can take up hundreds of MB.

So NiceEval trims as it writes to disk: any string in `events.json` or `trace.json` longer than 256 KiB keeps only its first 256 KiB, with one line of explanation at the end:

```text
[niceeval] truncated 51467156 → 262144 bytes
```

**This does not affect the verdict.** Assertions read the complete output at runtime; truncation only happens at the moment the file is written — the stored artifact is evidence, not an input to scoring. An assertion that should pass still passes, and one that should fail still fails.

To mark "something is missing here" faithfully in your own report, do not match the line of text above — read the structured field: truncated events and spans carry `truncated`, which holds the truncated position and the original byte count.

```typescript
const events = await attempt.events();
for (const e of events ?? []) {
  for (const t of e.truncated ?? []) {
    console.log(`${t.path} truncated, ${t.originalBytes} bytes originally`);
  }
}
```

This limit governs a **single string value**, not a whole JSON file. One file can hold many normal values; `diff.json` and source code are never truncated, because they have to keep their full semantics. So `.niceeval/` works well as the local source of truth but is not by default suitable for committing straight into Git. Before publishing, use `copySnapshots` below to pick artifacts and run the whole-file size check.

Truncation happens at the persistence boundary; it cannot stand in for the agent runtime limiting the tool output it sends to the model. If the runtime first stuffs a 50 MB tool result into a model request in full and gets a 413 back, NiceEval still records the attempt as `errored`; all this guarantees is that the post-failure events / trace are not blown up by that same output.

## Version: who wrote it, and can it be read

Every snapshot carries its own provenance: `producer` is the tool and version that wrote this result (niceeval itself, or a third-party harness converted through the write side), and `schemaVersion` is the on-disk format version. The format version only bumps on breaking changes, and the reader only accepts a matching version — **an incompatible on-disk version is never parsed, migrated, or guessed at**; the whole run goes into `skipped`:

```typescript
for (const s of results.skipped) {
  s.reason;                    // "incompatible-version" | "malformed" | "incomplete"
  s.schemaVersion;             // the format version that result declared
  s.producer;                  // { name, version }: who wrote it
}
```

`"incomplete"` is a vanishingly rare case: attempts are on disk but `snapshot.json` never got written — this can only happen in the tiny window where the snapshot directory was created but the process died before finishing the metadata, or someone manually deleted the file. It is not the same thing as the common case of "the process was interrupted": that produces an **unfinished snapshot** (`snapshot.json` is there, it's just missing `completedAt`), which reads fine — every completed attempt is visible — and only earns an `unfinished-snapshot` warning when `results.latest()` selects it (see below); it never lands in `skipped`.

An old version's results are never lost: they remain on disk as-is, viewable with the tool that wrote them — when `producer.name` is `"niceeval"`, suggest `npx niceeval@<producer.version> view` to the user; when it is a third-party harness, faithfully report its name and version rather than composing an incorrect npx command. `skipped` gives you exactly enough information to get this branch right.

## A snapshot is the directory itself

**A snapshot = one run of an experiment**, and physically it is exactly one snapshot directory (`.niceeval/<experiment>/<snapshot>/`) — there is no lower layer. A single `niceeval exp compare` invocation opens one independent snapshot directory per experiment involved; they are never merged, and nothing is aggregated across experiments on disk. "The latest per experiment" is therefore naturally snapshot-grained: if the whole `compare` group ran on Monday and only `compare/bub-gpt-5.4` was rerun on Tuesday, bub's latest snapshot lands on Tuesday while codex's stays on Monday, and `exp.latest` reflects each experiment's own history.

## Selecting snapshots: `results.latest()`

Most scenarios first answer "what's the latest result for each experiment right now" — the selector picks "the latest per experiment" for you, the same policy as the default report. It returns a **Scope**: snapshots, coverage facts, and warnings travel together:

```typescript
const latest = results.latest({
  experiments: "compare/",     // optional: experiment id prefix filter (string | string[]), same prefix semantics as the CLI
});

latest.snapshots;              // Snapshot[]: the latest per experiment
latest.coverage;               // ScopeCoverage[]: per-experiment coverage facts
latest.warnings;               // structured warnings, not pre-rendered text
```

"Latest" can be incomplete: rerunning just one question is a normal debugging move, and the latest snapshot it produces covers only that one question. The selector compares each selected snapshot's coverage against that experiment's historical union (`exp.evalIds`) and records the gap on `coverage`:

```typescript
latest.coverage[0];
// {
//   experimentId: "midterm/bub-gpt-5.4",
//   knownEvalIds: ["algebra/quadratic", …],   // denominator: every eval this experiment has ever seen
//   missingEvalIds: ["geometry/area", …],     // evals with no attempt under the current selection
// }
```

The gap is a per-eval fact, so it's rendered at the eval's own row: `niceeval show` / `view` turn a missing eval into a "no result under the current config" placeholder row in the leaderboard, next to a copy-pasteable rerun command — you see the denominator gap right in the table you're already looking at. Programmatic checks are just as direct: a CI check like "fail if coverage shrinks" tests `coverage.some((c) => c.missingEvalIds.length > 0)`, no text parsing needed. The gap is always computed, never silent.

`warnings` holds completeness problems that can't be pinned to a single eval: a selected snapshot that never wrapped up (interrupted process) goes into `unfinished-snapshot`, and an unreadable snapshot found while scanning the results root goes into `unreadable-snapshot`. Each carries a `kind`, judgable structured fields, and a pre-rendered English `message` — print it as-is if you want to display it.

`snap.diagnostics` and `warnings` answer different questions: warnings describe how this batch of data got picked, while `diagnostics` describes what went wrong during one particular experiment run — an experiment-level teardown failure, an unenforceable budget — things that can't be pinned to a single eval, so they get written out with the snapshot they actually belong to at wrap-up. Both stay anchored to their own `experimentId` and `startedAt`; they are never merged into a Scope-wide conclusion, nor copied onto every attempt. Display them with the `SnapshotDiagnostics` component, which keeps each entry's source snapshot identity and age intact.

Every attempt also carries **staleness**: a carried entry (a cache hit — last round's result folded forward into this snapshot) has `attempt.carried === true`, and the leaderboard marks those rows with `↩` plus a human time distance (e.g. `↩ 3d`) — old but valid, since the fingerprint guaranteed nothing changed, so it's a row-level annotation, not a warning. To see only what was actually run just now, use `results.latest({ fresh: true })` or the CLI's `--fresh`; excluded evals turn into placeholder rows instead of silently disappearing.

Scope is the common input for both the [report building blocks](/tutorials/custom-reports) and `copySnapshots` below: pass in a `Scope` and coverage and warnings ride along (placeholder rows and the `ScopeWarnings` component render them faithfully); a hand-picked `Snapshot[]` array is accepted just the same. Every entry in `scope.snapshots` is the real, persisted snapshot on disk, and `diagnostics` travels with it, so either input can render snapshot diagnostics. Fine-tuning the official policy does not require downgrading to a plain array: `latest.filter((s) => s.experimentId !== "compare/broken")` returns a new Scope — snapshots are trimmed, coverage and warnings are pruned down to the surviving experiments, and provenance is not lost. `filter` only ever trims down; **replacing** a pick ("swap in this experiment's previous complete snapshot instead") is not its job — go back to `exp.snapshots` and pick it yourself; a hand-picked array has no picking process, so naturally it carries no coverage facts or warnings, which is also honest.

## A real script: distributions aren't folding

"The distribution of shell-command counts per agent" cannot be drawn as a table — that is a distribution, not a folding. Traverse directly:

```typescript
import { openResults } from "niceeval/results";

const results = await openResults(".niceeval");
const points = [];
for (const exp of results.experiments) {
  for (const attempt of exp.latest.attempts) {
    const o11y = await attempt.o11y();
    points.push({
      agent: exp.latest.agent,
      eval: attempt.evalId,
      passed: attempt.result.verdict === "passed",
      shellCommands: o11y?.shellCommands.length ?? 0,
    });
  }
}
```

Even on this deepest path, the disk layout is never touched: path joining, existence checks, version filtering, and snapshot splitting are all absorbed by the library. To lay this distribution into a report page, wrap it in a dual-render component with [`defineComponent`](/tutorials/custom-reports).

One obligation when accumulating across snapshots: by default, NiceEval carries results from the previous round that already have a determined verdict (passed / failed) and whose eval code and config are unchanged forward into the new snapshot (`--force` reruns everything), so the same attempt can end up existing in more than one stored result. A carried-over entry is not a hollow shell: it carries the original snapshot's `startedAt` (its identity anchor) and `artifactBase` (a relative path to the original snapshot's attempt directory), and lazy loading falls back through a candidate order — first this snapshot's own attempt directory, then the directory `artifactBase` points at (faithfully returning `null` if that original snapshot was cleaned up); `ref` points at the snapshot the entry was carried into. All four fields of the identity key `(experimentId, evalId, attempt, startedAt)` are present in the data — the first two are direct fields on the attempt, and the sequence number and `startedAt` are on `attempt.result`. The reader faithfully reflects this duplication; before aggregating across snapshots, deduplicate with `dedupeAttempts` by identity key, keeping the copy in the newest snapshot — the report building blocks' compute functions already do this internally, so remember to do the same pass in your own scripts:

```typescript
import { dedupeAttempts } from "niceeval/results";

const all = exp.snapshots.flatMap((s) => s.attempts);   // accumulate across snapshots: the full history
const { attempts, warnings } = dedupeAttempts(all);      // duplicates merged in by resume are folded away
```

## Writing: `createResultsWriter`

The write side and the read side are two halves of the same set of types, and the signature itself proves the roundtrip: the reader's `attempt.result` is assembled from two parts — snapshot-level fields (experimentId / agent / model / startedAt / producer) come from one declaration in `writer.snapshot()`, and everything else is exactly the type of `writeAttempt`'s first argument; the second argument is the type of the handful of artifacts lazy loading can retrieve. "`writeAttempt`'s argument + `snapshot()`'s declaration = everything the reader reads back" is backed by the type merge — snapshot-level fields are not in the attempt argument's type, so there is no question of "whose value wins". Use it to convert results from another platform or a home-grown harness into NiceEval's format:

```typescript
import { createResultsWriter } from "niceeval/results";

const writer = createResultsWriter(".niceeval", {
  producer: { name: "my-harness", version: "1.0.0" },
});

const snap = await writer.snapshot({   // create the snapshot directory (exclusive create, retries on a name clash) + write snapshot.json
  experimentId: "compare/my-agent",
  agent: "my-agent",
  model: "gpt-5.4",
  startedAt: sourceRun.startedAt,      // required: the dedup identity key is anchored on it
});

for (const r of convertedResults) {
  await snap.writeAttempt(r.result, {  // write result.json (the authoritative verdict record, written once) + split out artifact files
    events: r.events,                  // second arg = artifacts, all optional; whichever is missing, the reader lazily returns null
    diff: r.diff,
  });                                  // splitting artifact files, computing paths, and backfilling refs all happen inside the library
}

await writer.finish();                 // backfills completedAt on every snapshot; there is no closing aggregation of any kind
```

`writer.snapshot()` is the mirror of the reading side's "experiment → snapshot" layering: converting multiple experiments means opening multiple snapshot directories, with snapshot-level metadata like experimentId / agent / model / startedAt declared once here rather than stuffed into every attempt; the optional `knownEvalIds` (the union of evals known for this experiment) is also declared here — it is the denominator for coverage detection, so when a conversion only covers part of the question set, declare the full set honestly and the downstream coverage warning can be computed (`copySnapshots` automatically backfills this field when publishing, see below). The converted directory is a standard results directory: `niceeval show` / `niceeval view` can view it directly, the report building blocks can compute on it directly, with no format documentation to consult; `producer` shows up verbatim on the reading side's `snap.producer`. **Every file is written exactly once** is the write side's core promise: `snapshot.json` is written as soon as the run starts and only ever gets `completedAt` backfilled at wrap-up; `result.json` and artifacts land as each attempt completes. An interrupted process only loses attempts that were still in flight — completed verdicts and artifacts are already on disk. Only the genuinely rare case of "attempts on disk but no `snapshot.json` at all" lands in `skipped("incomplete")`; an unfinished snapshot whose metadata is intact reads fine and only carries a warning.

## Publishing: `copySnapshots`

Copy selected snapshots into a publish directory, format-aware — only the specified artifacts, only the selected attempts:

```typescript
import { openResults, copySnapshots } from "niceeval/results";

const results = await openResults(".niceeval");
await copySnapshots(results.latest(), "site-data/run", {
  artifacts: ["sources", "events", "trace", "o11y"],   // diff is never truncated, and is not included by default;
});                                                     // every file to be published also goes through a 50 MiB precheck;
                                                        // o11y is only a few KB — if your report uses metrics that read
                                                        // o11y (turns, for example), bring it along, or they render as "—"
```

The first argument accepts a `Scope` or a hand-picked `Snapshot[]` — the same input convention as the report building blocks. The legal values for `artifacts` are `"events" | "trace" | "o11y" | "agentSetup" | "diff" | "sources"`; by default it carries the five kinds other than `diff`. It errors rather than silently overwriting when the target directory already exists and is non-empty — if a publish script needs to be idempotent, clear the target directory first yourself.

Before copying starts, NiceEval plans every target file and checks its serialized size. If any single file exceeds the fixed 50 MiB limit, the whole copy fails before the target directory is created, and the error lists the path, the actual size, and what to do about it. You can exclude that class of evidence from `artifacts`; if the oversized events / trace were left behind by an older version, rerun with the current version and then publish. This check covers both source code / diffs, which have no per-value truncation, and JSON whose individual values are all fine but whose total is too large — so you don't discover a Git host's single-file limit only at `git push` time.

The size precheck only decides whether the whole copy succeeds or fails; it never deletes content out of the middle of an oversized file. Copying is faithful to the source: artifacts are copied byte-for-byte, never re-serialized or rewritten. The one thing automatically backfilled along the way is the **coverage fact at pick time**: the denominator of a coverage gap is an experiment's historical union, and a publish directory has no history — so every copied snapshot carries `knownEvalIds` (the union of evals known for that experiment at copy time), and the reader folds it into the computation of `exp.evalIds` (taking the union of local history and the value carried by the snapshot). Running `openResults().latest()` again on the publish directory recomputes the coverage gap through the same mechanism, without relying on the publisher's own account. The copied directory is a standard results directory — `niceeval view --results <dir>` views it directly; for wiring up the report site to update automatically on push, see [Publish a Report Through CI](/tutorials/publish-report).

## Layer overview

| Layer | Entry point | Answers |
| --- | --- | --- |
| The two official doors | `niceeval show` / `niceeval view` | View the official layout, zero code |
| Report building blocks | [Custom Reports](/tutorials/custom-reports), [Report Components](/reference/report-components) | Your own scoring and layout |
| Results Data API (this page) | `niceeval/results` | Algorithms folding can't express, feeding your own systems, reading and writing the format itself |

Each layer is built on the one beneath it, and the same on-disk artifacts are the single source of truth — derived output at higher layers can be deleted and recomputed at any time.

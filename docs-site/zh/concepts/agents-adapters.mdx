在 NiceEval 里，`Agent` 是 NiceEval 调用的抽象，`Adapter` 是具体实现。这一页讲清楚三件事：**两者的边界在哪、Adapter 唯一要实现的 `send` 函数传入什么返回什么、`t` 上的能力从哪来。**看完这页再去[写 send](/zh/concepts/write-send)——那里从三个问题出发，一路推导到两个完整的 Adapter 示例。

## 两个词，一个边界

- **Agent**：NiceEval 眼里的被测对象。一个有名字、一个 `send` 方法、外加几个所用零件的对象。运行器只跟这个接口打交道，从不写 `if (agent === "claude-code")` 这样的分支。
- **Adapter**：实现这个接口的代码，由你写（或用内置的）。它知道怎么鉴权、怎么调你的应用、怎么把返回翻译成标准事件流。

一次 `t.send()` 的完整往返，穿过这条边界两次：

```text
evals/*.eval.ts           NiceEval 运行器             Adapter（你写）             你的应用（一行不改）

await t.send("...")  ──▶  组装 TurnInput + ctx  ──▶  send(input, ctx)  ─────▶   前端在用的那个接口
t.reply              ◀──  折叠 events 成事实    ◀──  原始返回翻译成 events ◀──  响应 / SSE 流
t.calledTool()...         更新 t.reply 与用量        return Turn
```

去程：eval 里的驱动动词（`t.send` / `t.sendFile` / `t.respond`）到运行器都变成同一件事——组装 `TurnInput` 和本会话线的 `ctx`，调一次 Adapter 的 `send`。回程：Adapter 把应用的原始返回翻译成 `Turn`——其中 `events` 是一个按发生顺序排列的事件对象数组（下文有实际值）——运行器折叠这个数组，断言和评分读的全是折叠后的事实。边界两侧互不知晓对方细节：运行器不知道 HTTP 长什么样，Adapter 不知道 eval 里写了哪些断言。

被测系统的 URL、鉴权、协议细节都是 Adapter 的私有配置（读环境变量或闭包），NiceEval 永远看不到。experiment 按名字引用 Agent；experiment 名之后的位置参数只选择"跑哪些 eval"。

```bash
# 同一份 eval，切换不同 Adapter；URL 是 Adapter 的私事
npx niceeval exp local support
npx niceeval exp prod support
```

## 接入分三个 Tier

按「要不要动应用代码、动到什么程度」，接入分三档（每档买到什么、怎么升级，见[接入分三个 Tier](/zh/guides/connect-your-agent#接入分三个-tier)）：

- **Tier 1（只接 send）**：应用代码一行不改，接入只有一个 Adapter 文件。文本断言、judge、多轮、HITL、手写映射的工具断言都在这档；应用接口暴露模型选择的话，模型对比 experiment 也在（`experiment.model` 经 `ctx.model` 透传）。
- **Tier 2（send + OTel）**：同一个 send，加上应用把 OTel span 发给 NiceEval——事件从 span 派生、免手写映射，还有 trace 瀑布图（见[通过 OTel 接入](/zh/guides/connect-otel)）。已埋点的应用零改动。
- **Tier 3（侵入改造 + experiment flags）**：改应用内部代码，把 prompt / 工具集 / feature 变体暴露成 experiment 可传的 `flags`（经 `ctx.flags` 到 Adapter、随请求交给应用），做 feature A/B test。

前两档都是**无侵入**的：应用由你按它自己的方式启动（`pnpm start`、部署在哪都行），eval 侧不 spawn 应用进程、不另开端口——Adapter 只对着**用户前端本来就在用的那个接口**（HTTP 端点、SSE 流）收发，连不上时报"先起应用"的明确错误。观测数据走应用已有的返回信息或 OTel。Tier 3 虽然改了应用内部，Adapter 依然只经对外接口调用应用——侵入的是应用（把变体暴露成配置），不是接入面。

这意味着 Adapter 是 eval 与被测应用之间**唯一的接触面**。eval 侧的驱动 API——`t.send()`、`t.sendFile()`、`t.newSession()`、HITL 的 `t.respond()` / `t.respondAll()`——不管看起来多不一样，穿过运行器到达 Adapter 时**全部收敛成"再调一次 `send`"**。怎么收敛、send 里怎么接，见[写 send](/zh/concepts/write-send)。

## send 函数：传入什么，返回什么

不管 Adapter 连的是 HTTP 服务还是沙箱里的 CLI，暴露给运行器的接口完全一致：

```ts
interface Agent {
  readonly name: string;
  send(input: TurnInput, ctx: AgentContext): Promise<Turn>;
  readonly session?: ServerSession | ClientHistory<unknown>;  // 会话件：有件 = 支持多轮（见「能力从哪来」）
  // 可选：setup / teardown（每沙箱一次的生命周期）；
  // 观测相关的可选成员不参与 send，见「通过 OTel 接入」
}
```

`send` 是唯一要实现的函数。它的签名里只有三个类型，分开看。

### 传入：`TurnInput`

```ts
interface TurnInput {
  readonly text: string;                          // 每轮必有：发送的文本；回答轮 = 回答文本（多条以换行连接）
  readonly files?: readonly InputFile[];          // 仅 t.sendFile 轮：附带的文件（base64）
}

interface InputFile {
  readonly filename?: string;     // 可选，供 adapter / 模型参考
  readonly mimeType: string;      // 如 image/png
  readonly dataBase64: string;    // base64 内容，JSON 友好，可直接放进请求体
}
```

eval 侧不管调的是 `t.send()`、`t.sendFile()` 还是 `t.respond()`，到 Adapter 时都是一次普通的 `send`，差别全在字段上：

| eval 侧 | `text` | `files` |
|---|---|---|
| `t.send(text)` | 发送文本 | — |
| `t.sendFile(path, text?)` | 附言（可为空串） | 有 |
| `t.respond(...)` / `t.respondAll(...)` | 回答文本（多条按 `"\n"` 拼接） | — |

`files` 接口不收文件就忽略。

#### 回答怎么到达

HITL 回答轮里，人的裁决以**纯文本**经 `input.text` 到达：`t.respond(...answers)` 把多条回答按 `"\n"` 拼接；`t.respondAll(optionId)` 给每条待处理请求各生成一条相同的回答再拼接。eval 侧只校验"确实有待处理请求"——回答内容本身不校验，`"approve"` 拼错会原样传给应用。三种典型形态：

```ts
// ① 单个待处理请求，回答命中选项（approve / deny 同理）
await t.respond("approve");
// → input.text = "approve"

// ② 回答不是任何选项 → 自由文本（等补充信息的请求就是这种）
await t.respond("收件人改成 ceo@corp.com");
// → input.text = "收件人改成 ceo@corp.com"

// ③ respondAll：每条待处理请求一条相同回答（两条请求并停时）
await t.respondAll("approve");
// → input.text = "approve\napprove"

// 多个请求并停、要给不同答案时，按你吐出 input.requested 的顺序逐条给：
await t.respond("approve", "deny");
// → input.text = "approve\ndeny"
```

Adapter 侧的对应义务：回答按你上一轮吐出 `input.requested` 事件的顺序对位（挂起时把待裁决的 `callId` 一并存进会话状态，回答轮取出配对）；被人拒绝的调用把 `action.result` 的 `status` 置 `"rejected"` 而不是 `"failed"`——拒绝是人的决定、不是工具故障，这样 `noFailedActions()` 不误伤，`calledTool(..., { status: "rejected" })` 可精确断言。

### 上下文：`AgentContext`

```ts
interface AgentContext {
  // 每轮都在的三个
  readonly signal: AbortSignal;    // 运行器的超时与取消：透传给你发出的每个请求
  readonly session: AgentSession;  // 本条会话线的状态槽
  log(msg: string): void;          // 写进本轮日志，niceeval view 里可见

  // 按接入档位出现的三个（见「接入分三个 Tier」）
  readonly model?: string;         // Tier 1 的模型对比：experiment.model 透传；应用接口收模型选择就转发
  readonly telemetry?: Telemetry;  // Tier 2：本次运行的 OTLP 接收信息（endpoint / env / headers）
  readonly flags: Readonly<Record<string, unknown>>;  // Tier 3 的 feature A/B：experiment.flags 透传；没配时是 {}

  readonly sandbox?: Sandbox;      // 仅 defineSandboxAgent 构造的 Agent 会拿到
}

interface AgentSession {
  readonly state: Record<string, unknown>;  // 每条会话线一份的自有状态槽，起始 {}
}
```

`ctx` 里没有要你检查的标志位。三个档位字段都是"透传"语义：`model`、`flags` 由 experiment 声明、运行器原样递过来，Adapter 只负责随请求转发给应用，不解释它们的含义；`telemetry` 仅在声明了 tracing / OTel 接入时出现，用法见[通过 OTel 接入](/zh/guides/connect-otel)。

`session.state` 是本条会话线的自有状态槽，NiceEval 对它只承诺一件事：**同一条会话线的每次 `send` 拿到同一份 `state`，新会话线（eval 的第一轮，或 `t.newSession()` 之后）拿到一份空的。**它从 `{}` 起步，框架从不往里写数据——里面只会有你（或你用的官方会话件）存过的东西。用官方件时这份状态你连碰都不用碰，件会替你存取。

### 返回：`Turn`

```ts
interface Turn {
  readonly events: StreamEvent[];          // ★ 标准事件流——每个 Adapter 的核心产物
  readonly data?: unknown;                 // 本轮结构化产物（应用有结构化输出时才填），供 outputEquals / outputMatches 断言
  readonly status: "completed" | "failed" | "waiting";  // waiting = 停轮等人（HITL）
  readonly usage?: Usage;
}
```

`events` 就是一个**普通的 JS 数组**：这一轮里发生的每件事一个对象，按真实发生顺序排列。比如"北京今天多少度?"这一轮，agent 先查天气再回答，`events` 长这样：

```ts
[
  { type: "action.called", callId: "c1", name: "get_weather", input: { city: "北京" } },
  { type: "action.result", callId: "c1", output: { temp: 21 }, status: "completed" },
  { type: "message", role: "assistant", text: "北京今天 21°C。" },
]
```

对象统共九种类型（`message`、`action.*`、`input.requested`……完整清单见[事件流参考](/zh/reference/events)）。断言读的就是这个数组：`t.calledTool("get_weather")` 数 `action.called`，`t.reply` 取最后一条 assistant `message`。注意 `send` 每次只返回**本轮**的数组——跨轮拼成整条会话线是运行器的事，见下一节。多数情况下你也不手写这些对象：官方转换器的返回值就是一个填好 `events`、`usage`、`status` 的完整 `Turn`，怎么选转换器见[写 send](/zh/concepts/write-send)。

`data` 不是"随便放点什么"的口袋：它是**本轮的结构化产物**——应用本来就有结构化输出（JSON 回复、`experimental_output` 这类）时，Adapter 把它放进 `data`（比如 `JSON.parse(reply.content)`，或 `fromAiSdk` 配 `data: result.text`），eval 侧用 `turn.outputEquals(value)`（深相等）或 `turn.outputMatches(schema)`（Standard Schema / zod 校验）断言它：

```ts
// eval 侧：data 的形状用 schema 断言，结构化语义归应用自己
const turn = await t.send("这张发票总额多少?");
turn.outputMatches(z.object({ amount: z.number(), currency: z.string() }));
```

`data` 的类型是 `unknown`——要在 eval 代码里直接读字段，自己收窄。只回文本的应用永远不碰这个字段，不要把原始响应 body 复制进去凑数。

### send 返回之后：Turn 的四个字段各去哪

`send` 一 return，Adapter 的工作就结束了——剩下全是运行器的事。四个字段各有明确去向：

| Turn 字段 | 运行器拿它做什么 | eval 作者在哪感知到 |
|---|---|---|
| `events` | 追加进本会话线的事件流，`deriveRunFacts` 折叠成事实（`toolCalls`、`parked`、`messageCount`…）；最后一条 assistant `message` 同时更新 `t.reply` | 全部作用域断言：`t.calledTool()`、`t.messageIncludes()`…；`t.events` 可直接查 |
| `status` | 记录本轮终态；`"waiting"` 时把本轮的 `input.requested` 收进待答列表 | `t.succeeded()` / `t.parked()`；`"waiting"` 之后用 `t.respond()` 接 |
| `usage` | 逐轮累加到会话线与整次运行 | `maxTokens` / `maxCost` 评分器、报告里的用量 |
| `data` | 原样挂到本轮句柄上 | `turn.data`（`unknown`，自行收窄）、`turn.outputEquals()` / `turn.outputMatches()` |

`await t.send()` 拿到的句柄就是对这一轮 Turn 的视角：turn 级断言（`draft.parked()` 这类）只看这一轮的 `events`，`t` 级断言看整条会话线累计的。所以 Adapter 不需要"配合"任何断言——把这四个字段填对，断言、评分、报告全部在下游自动发生。

## 能力从哪来：优先由构造派生，声明要诚实

能力位是 `defineAgent` 上的显式声明（`capabilities: { toolObservability: true }` 这样的布尔字段），但 NiceEval 尽量不让你填问卷——**能派生的能力由构造派生**：你递了什么件、吐了什么事件、走了什么构造函数，`t` 上就解锁什么：

| `t` 上解锁什么 | 怎么解锁 | 你写什么 |
|---|---|---|
| `t.send`（第一轮）、`t.sendFile`、`t.check`、`t.judge`… | 任何 Agent 都有 | 无 |
| 第二轮起的 `t.send()`、`t.newSession()` | `conversation` 能力——定义里递了 `session` 件（`clientHistory` / `serverSession`）自动为真 | 把本来就在用的件递给 `defineAgent` |
| `t.respond()` / `t.parked()` 等 HITL | send 返回过 `"waiting"` + `input.requested` 事件 | 无——做到了就是有 |
| `t.calledTool()` / `t.notCalledTool()` / `t.toolOrder()` 等工具断言 | `toolObservability` 能力 | 一行诚实声明：确认事件流对工具调用无遗漏时写 `capabilities: { toolObservability: true }` |
| `t.sandbox`、`t.fileChanged()` 等 | `defineSandboxAgent` 构造（默认带 `workspace` / `sandbox` 等全套能力位） | 无 |
| `EvalResult.trace`、`niceeval view` 瀑布图 | `capabilities: { tracing: true }` + `tracing` 块 / `events: otelEvents()` | 声明 + 本来就要写的块 |

工具断言那行是唯一要你亲手声明的：`notCalledTool` 断的是"没发生"，前提是"事件全量"——这只有事件来源有完整性契约时才成立（SDK 原生事件流透传、AI SDK 的 `result.steps`、Responses 的 `output`）。所以远程 / 进程内 agent 的能力位**默认全空**：没声明 `toolObservability` 时，`calledTool` / `notCalledTool` 这族断言一被调用就被能力守卫拦下、报清晰错误，而不是对着不完整的事件流假通过（负断言假通过是最危险的失真）。声明要诚实：接口只回最终答案却声明了 `toolObservability`，守卫拦不住你，失真就是你自己签的字。逐能力的精确义务见[能力参考](/zh/reference/capabilities)。

## 另一种 transport：sandbox Agent

评 claude-code / codex / bub 这类要改文件的 coding agent CLI 时，用 `defineSandboxAgent`：契约完全相同（同一个 `send`、同一个 `Turn`），差别只在"连接"不是 HTTP 而是"在沙箱里 spawn CLI、读回 transcript"，运行器经 `ctx.sandbox` 把沙箱句柄传进来。三个内置 Agent 开箱即用，各自做到哪些能力见[内置 Agent 能力参考](/zh/reference/builtin-agents)；自己接新 CLI 见[沙箱 Agent 指南](/zh/guides/sandbox-agent)。

## Agent × Sandbox 正交

`experiment.agent` 选择谁来执行任务，`experiment.sandbox`（`dockerSandbox()` / `vercelSandbox()` / `e2bSandbox()`）选择在哪里执行；`niceeval.config.ts` 里的 `sandbox` 是项目级兜底（`exp.sandbox ?? config.sandbox`）。任何 sandbox Agent 配任何沙箱后端；Tier 1 的无侵入 Adapter 完全用不到 sandbox。没有对应的 CLI flag，也没有自动探测——sandbox Agent 的 experiment（或兜底的 config）必须显式设置 `sandbox`，否则创建时直接报错。

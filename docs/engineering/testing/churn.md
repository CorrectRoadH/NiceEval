# 测试跟改率 —— 用 git 历史定位不稳定测试

好测试的硬指标是**改功能时测试的改动要少**：[变更预算](README.md#变更预算无关测试变红是缺陷)规定一次改动允许触碰的测试 = 契约 diff 的影响面，实现重构变红的测试是缺陷。本篇给这条规则一个可量化的回溯口径——**测试跟改率**：过去改功能（src 变更）时，哪些测试文件老是跟着一起改、改了多少。跟改率高的测试就是变更税的主要征收点，按缺陷排查：要么锁定了实现细节（改写断言到契约层），要么站错了层（按[分层](README.md#分层谁负责证明什么)挪去 E2E）。

## 口径

对每个测试文件统计两个数：

- **跟改次数**：同一个 commit 里既改了 `src/` 非测试文件、又改了该测试文件的 commit 数。这是"改功能连带改测试"的直接证据。
- **总变更次数**：该测试文件被任何 commit 触碰的次数。

跟改次数是排查的主排序键；跟改次数 / 总变更次数的比值接近 1 说明该测试几乎从不独立演进、纯粹被实现拖着走。契约真变了导致的合法跟改也计入——口径不区分动机，区分动机是排查阶段对照该文件所属 `cases.md` 与当期契约 diff 的人工判断：登记行同批改了的是合法影响面，登记行没动测试却动了的就是不稳定测试。

## 度量命令

在仓库根执行（窗口默认取最近六个月，按需调整）：

```sh
git log --since="6 months ago" --format='@%H' --numstat -- src test \
| awk '
  /^@/ { if (srcTouched) for (f in tests) count[f]++; delete tests; srcTouched = 0; next }
  NF >= 3 {
    f = $3
    if (f ~ /\.test\.(ts|tsx)$/) { tests[f] = 1; total[f]++ }
    else if (f ~ /^src\//) srcTouched = 1
  }
  END {
    if (srcTouched) for (f in tests) count[f]++
    for (f in count) printf "%4d  %4d  %s\n", count[f], total[f], f
  }
' | sort -rn | head -30
```

输出三列：跟改次数、总变更次数、测试文件。已知局限：rename 在 `--numstat` 里以 `{old => new}` 形态出现、会被当成新路径重新计数；批量重构类 commit（如目录搬移）会给全体测试各记一次——排查时看清单头部的相对差距，不看绝对值。

## 排查动作

1. 取排行头部（跟改次数断崖以上）的文件，逐个对照其 `cases.md`：该文件历次变更里，有多少次对应登记行同批变更？
2. 登记行没动、测试却反复动的：按[无意义或脆弱的 unit](unit/README.md#无意义或脆弱的-unit) 归类病因——锁实现细节、断言渲染产物、mock 复述——改写或按分层挪去 E2E。
3. 改写后把病因记入 `memory/`（修法台账），让同类测试不再以同样方式被写出来。
4. 重大重构（如测试体系重划）后重置观察窗口，用新窗口的排行验证改写有效——头部文件应该换人，而不是同一批测试换个姿势继续跟改。

这套口径是回溯诊断，不是 CI 门禁：跟改率天然滞后且含合法变更，做成红绿灯只会逼人拆 commit 绕过。它的消费场景是周期性（或大重构前后）人工跑一次，为「哪些测试该重写」提供证据排序。

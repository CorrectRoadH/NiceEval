In NiceEval, an `Adapter` is the layer between the runner and the system under test. It implements `send`: take an eval-side action such as `t.send()`, call your application or CLI, and translate the result into a standard `Turn`.

Read [Write Send](/guides/write-send) for the implementation tutorial.

## Adapter boundary

If the subject under test uses a standard protocol, use an official adapter or converter. If it has a custom HTTP, gRPC, WebSocket, SDK, or CLI surface, write an adapter. The adapter owns auth, transport, protocol details, session continuation, and event mapping.

![Full round trip of t.send: eval calls t.send, the runner builds TurnInput and ctx, the adapter calls your app and returns a Turn with standard events.](/images/agent-turn-roundtrip-en.svg)

The app URL, credentials, and protocol options belong in the adapter factory. `defineExperiment` receives an already configured agent instance. Per-turn dynamic values such as `ctx.model`, `ctx.flags`, and `ctx.telemetry` are passed through each `send` call.

```ts
// agents/web-agent.ts
import { defineAgent } from "niceeval/adapter";
import type { Agent } from "niceeval/adapter";

interface WebAgentOptions {
  baseUrl: string;
  apiKey?: string;
}

export function webAgent(options: WebAgentOptions): Agent {
  const baseUrl = options.baseUrl.replace(/\/$/, "");

  return defineAgent({
    name: "web-agent",
    async send(input, ctx) {
      const response = await fetch(`${baseUrl}/api/turn`, {
        method: "POST",
        headers: {
          "content-type": "application/json",
          ...(options.apiKey ? { authorization: `Bearer ${options.apiKey}` } : {}),
          ...ctx.telemetry?.headers,
        },
        body: JSON.stringify({
          message: input.text,
          files: input.files,
          model: ctx.model,
          flags: ctx.flags,
        }),
        signal: ctx.signal,
      });

      if (!response.ok) {
        return {
          status: "failed",
          events: [{ type: "error", message: `HTTP ${response.status}` }],
        };
      }

      const body = await response.json();
      return {
        status: "completed",
        events: body.events,
        data: body.data,
        usage: body.usage,
      };
    },
  });
}
```

## Integration tiers

NiceEval integrations are grouped by where the adapter connects and what extra observability it receives:

- **Tier 1: send only.** Your app code can stay unchanged. The adapter calls your existing interface and maps the response.
- **Tier 2: send + OTel.** The app sends OpenTelemetry spans to NiceEval. Events and the trace waterfall can be derived from spans.
- **Tier 3: application changes + experiment flags.** The app exposes feature switches through `ctx.flags` for A/B evaluation.

See [Tier](/concepts/tier).

## `send`

Every adapter exposes the same interface, regardless of whether it calls an HTTP service or a CLI in a sandbox:

```ts
interface Agent {
  readonly name: string;
  send(input: TurnInput, ctx: AgentContext): Promise<Turn>;
}
```

### Input: `TurnInput`

```ts
interface TurnInput {
  readonly text: string;
  readonly files?: readonly InputFile[];
  readonly responses?: readonly InputResponse[];
  readonly outputSchema?: JsonSchema;
}
```

`t.send()`, `t.sendFile()`, `t.respond()`, and structured-output sends all become ordinary `send` calls. The difference is in the fields:

| Eval-side call | `text` | `files` | `responses` | `outputSchema` |
|---|---|---|---|---|
| `t.send(text)` | Sent text | - | - | Present only when `output` is declared |
| `t.sendFile(path, text?)` | Caption, maybe empty | Present | - | Same |
| `t.respond(...)` / `t.respondAll(...)` | Human answer text | - | One structured answer per request | Same |

If your app does not accept files or schemas, ignore those fields. Output validation still happens in the runner.

### Context: `AgentContext`

```ts
interface AgentContext {
  readonly signal: AbortSignal;
  readonly session: AgentSession;
  log(msg: string): void;

  readonly model?: string;
  readonly telemetry?: Telemetry;
  readonly flags: Readonly<Record<string, unknown>>;
  readonly sandbox?: Sandbox;
}
```

`model`, `flags`, and `telemetry` are pass-through values. The adapter forwards them to the app when the app understands them. `session` is the state slot for one conversation line. The same line gets the same `ctx.session`; `t.newSession()` gets a fresh one.

### Return: `Turn`

```ts
interface Turn {
  readonly events: StreamEvent[];
  readonly data?: JsonValue;
  readonly status: "completed" | "failed" | "waiting";
  readonly usage?: Usage;
}
```

`events` is the standard event stream, ordered by when things happened. Assertions read this stream: `t.calledTool("get_weather")` counts `action.called`; `t.reply` reads the last assistant `message`.

`data` is only for structured output declared by the eval. If a turn declares `output`, the adapter puts the parsed object in `data` and the runner validates it. If no `output` was declared, do not copy raw response bodies into `data`.

## After `send` returns

| Turn field | Runner behavior | Eval author sees |
|---|---|---|
| `events` | Appended to the session event stream and folded into facts | `t.calledTool()`, `t.messageIncludes()`, `t.events`, `t.reply` |
| `status` | Records the turn outcome; `"waiting"` collects pending `input.requested` events | `t.succeeded()`, `t.parked()`, `t.respond()` |
| `usage` | Accumulates into run and session usage | `maxTokens`, `maxCost`, reports |
| `data` | Validated against the send `output` schema | Strongly typed `turn.data`, `outputEquals` |

## Where capabilities come from

Capabilities are proof by construction, not a questionnaire.

| What unlocks on `t` | Evidence | What you write |
|---|---|---|
| `t.send`, `t.sendFile`, `t.check`, `t.judge` | Every agent has them | Nothing |
| Multi-turn, `t.reply`, `t.newSession` | You connected `ctx.session` history or id/capture | Usually a `conversation` capability declaration |
| `t.respond`, `t.parked` | A turn returned `"waiting"` plus `input.requested` | Nothing |
| Positive tool assertions | `events` contain `action.*` | Events |
| Trustworthy negative tool assertions | Official converters or a verified complete mapping | `toolObservability` only when you can prove it |
| `t.sandbox`, file assertions | `defineSandboxAgent` | Nothing |
| Trace waterfall | `tracing` block or `events: otelEvents()` | The tracing source itself |

See [Capabilities](/reference/capabilities).

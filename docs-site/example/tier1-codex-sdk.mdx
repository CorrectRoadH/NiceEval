Compare:

- **before**: [`examples/zh/origin/codex-sdk`](https://github.com/CorrectRoadH/niceeval/tree/main/examples/zh/origin/codex-sdk) - a standalone `@openai/codex-sdk` HTTP service.
- **after**: [`examples/zh/tier1/codex-sdk`](https://github.com/CorrectRoadH/niceeval/tree/main/examples/zh/tier1/codex-sdk) - the same app with NiceEval adapter, evals, and experiments.

**Integration style**: Codex native OTLP plus adapter-side message completion. `otelEvents({ dialects: [otel.codex] })` derives tool names, call ids, and usage from Codex spans. The adapter fills message text from the app stream where spans do not carry it.

## File map

```text
codex-sdk/
├── niceeval.config.ts
├── agents/
│   └── codex-sdk.ts
├── evals/
│   ├── create-file.eval.ts
│   ├── run-command.eval.ts
│   └── weather-tool.eval.ts
└── experiments/
    └── assistant.ts
```

## What to inspect

- `agents/codex-sdk.ts`: how Codex spans and app-stream messages are merged.
- Coding-task evals: file creation and command execution are asserted through the normal event and output surface.
- `niceeval.config.ts`: OTel receiver setup used by the app during eval runs.

## Run it

```bash
cd examples/zh/tier1/codex-sdk
pnpm install
pnpm dev
npx niceeval exp assistant
npx niceeval view
```

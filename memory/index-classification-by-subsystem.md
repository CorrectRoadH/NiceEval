# 设计裁决：memory 索引按子系统主轴归档,类型降为大区子标题

- **日期**：2026-07-21
- **原问题**：`memory/INDEX.md` 长到 ~230 条时扫不动,根因不是条目多而是**分类轴串味**——大部分分区按子系统(沙箱/judge/o11y/CLI/examples/docs/环境),但「设计决定」一个分区按**体裁**(裁决 vs 台账)单列。结果同一子系统的裁决一半散在子系统分区、一半堆在「设计决定」里,**同一块工作要扫两个地方**。两堵最高的墙是「设计决定」(~50)和「沙箱」(~42);「o11y 采集与 view」把报告 web 面和 o11y 采集两个主题焊在一起。
- **裁决**：单一「子系统」主轴(分区=「动手前扫哪块」)。① 溶解「设计决定」分区,裁决按它约束的子系统归位;真正跨切面的(术语大改名/测试预算/测试体系/错误反馈契约/轮标签)进新的「跨切面裁决」区。② 「报告 · view」从 o11y 拆出独立成区(最大单一主题),「o11y 采集」收窄为 parser/span/event/execution-tree。③ 沙箱/Runner/报告三个大区内部拆「裁决 / 台账」两个子标题——**台账≈bug**(现象/根因/修法),**裁决≈DX/设计反馈**(裁决/曾选方案/否决理由/日期);小区(o11y/写 eval/docs/环境/examples,均 ≤20 条)平铺不拆。④ 顶部维护规则同步重写描述新归档法。落点 commit `05a040e`(232 条一条不减)。
- **曾选方案**：把「bug / DX 反馈」当**顶层主轴**(先分 bug 与设计反馈两大类,再分子系统)。**否决理由**:召回模型是「动手涉及某块前先扫该块」,类型轴切在顶层会让同一子系统的工作永远扫两处——这**正是原问题的根因**,不能用同一个反模式去修。类型只降为大区内的子标题(裁决/台账),既让 DX/bug 可分辨,又不破坏子系统主轴。
- **同场否决**:把「已修」条目分离到单独文件(如 `INDEX-fixed.md`)以缩短可见长度——违反「不归档不删除」规则(已修条目是复盘材料);缩短靠子标题分组与大区拆分实现,不靠隐藏条目。
- **守护**:`test/memory-index.test.ts` 只查每条有索引行,重排不动它;归档纪律靠 INDEX.md 顶部维护规则(相关:[[design-status-from-docs-not-src]] 是同类「查询纪律升格为规则」的先例)。

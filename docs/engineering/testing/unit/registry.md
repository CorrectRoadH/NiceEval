# 覆盖登记 —— 先声明，后写测

每篇 Feature 测试文档（清单见 [unit/README.md](README.md)）的「覆盖规范」小节是该 Feature 契约影响面的登记面（体系定位见[测试体系总纲](../README.md)）。本篇定义它的著作权、写测流程和预算：覆盖规范是测试数量的闸门，不是写完测试之后的台账。

## 著作权：覆盖类别是设计产物

覆盖类别与 Feature 契约同批定稿。设计迭代重写 `docs/feature/` 受影响小节时，同批重写对应测试文档的覆盖规范条目——交付物是「这条契约由哪些类别的行为证明、fixture 要什么区分力、哪类错误不许静默放走」，不是逐场景表格，更不是测试代码。**具体场景由测试代码枚举：测试名就是场景清单，docs 不复述**。

这个顺序把测试数量的裁决放到最便宜的评审面上：拒掉一条覆盖类别，比删掉一批已经写好的测试便宜一个数量级。

## 流程：测试只实现已声明的类别

- 实现阶段的测试工作是**把已声明的覆盖类别变绿**：每条测试要能指认自己实现所属文档的哪条类别，测试名描述契约和场景。指认不了的测试在变更预算之外，按[核心判据](README.md#核心判据)默认删。
- 实现中途发现的新行为类别，先补文档条目、按契约影响面评审，再写测试。补一条类别是一句 diff，它把「要不要多这类测试」的裁决从代码评审前移到文档评审。
- 修 bug 的测试要能追溯台账：bug 修复（先确认所属覆盖类别、缺类别先补，再写红测试、修实现）产生的测试在测试处加一行 `// bug: memory/<条目>.md` 引用对应 memory 条目。memory 是修法台账，复盘「这个修法合理不合理」时要能从测试反查现象与根因，也要能从台账找到锁定它的测试；引用的条目真实存在由仓库守护校验。

## 预算：类别内按等价类，不按数量

- 一条类别覆盖会改变语义的等价类、边界值与状态组合。每个新增场景必须有区分力——让契约算法与某个常见错误算法在该输入上给出不同答案（规则见 [Harness](harness.md)）；给不出这个区分，新增场景就是既有场景的复述。
- **同一场景的第二条测试是维护负担，不是更多覆盖**——与 [E2E 的仓库 Eval 预算](../e2e/adapter/README.md)同一条纪律。
- 没有新增契约的改动（实现重构、性能优化）零新增测试是合格状态，不是偷懒；「顺手多补几条」不是勤奋，是给未来每一次契约变更加税。
- 同一不变量要跨故障边界各验证一次（如领域规则层一次、公共组合层一次）时，两处各自写明自己放走的错误类别不同；写不出不同的错误类别，就只保留一处。

## 与守护的分工

机器守护（`test/cases-registry.test.ts`，见 [README · 套件边界与仓库守护](README.md#套件边界与仓库守护)）只保证测试套件与测试文档不整册脱钩：src 测试文件头注声明所属文档且指向真实文件、每篇 Feature 测试文档至少被一个测试声明、`// bug:` 引用真实存在。类别级对应靠两条评审义务：测试改动对照覆盖规范核对影响面没有越界；覆盖类别改动对照契约 diff 核对没有超出影响面。

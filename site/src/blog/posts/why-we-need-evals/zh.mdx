## 问题不是 agent 会失败

Agent 会失败，这件事本身并不意外。真正的问题是，它们经常以很难分类的方式失败。

模型可能带着过期上下文自信作答。它可能在错误的时机调用了正确的工具。它可能完成了看得见的任务，同时破坏了一个隐性的要求。它也可能在一次 happy path demo 里表现很好，但同一条 workflow 换一个稍微不同的 fixture 就崩掉。

如果我们手上只有 transcript 和感觉，每一次回归都会变成争论。有人觉得这个 agent 已经够好了，有人看到一个细微但真实的行为断裂。没人能确定下一次模型、prompt、adapter 或 sandbox 的变化到底让系统变好了还是变坏了。

> eval 是把 agent 行为从争论变成工程信号的方法。

## 单元测试不够

传统测试仍然必要。它们告诉我们代码路径能不能跑通、schema 是否对齐、集成返回的形状是否符合预期。

但 agent 行为不只是代码行为。它包含判断、时机、工具选择、记忆、grounding 和恢复能力。这些东西需要能观察整次运行的检查：

- assistant 是根据上传文件回答，还是在猜？
- 它有没有在追问里保留上下文？
- 当答案已经在上下文里时，它有没有避免不必要的工具调用？
- 这次运行失败，是 agent 真错了，还是 adapter 崩了？

最后这个区分很重要。运行时失败不是模型质量信号。真实断言失败才是。

## 好的 eval 描述产品行为

最好的 eval 不是冷知识题，而是小型产品 fixture。

对 NiceEval 来说，这意味着 eval 应该描述我们真的关心的行为：support agent 是否基于检索上下文回答，coding agent 是否编辑了正确文件，多模态 agent 是否根据收到的图片作答，或者一个 service 是否在 retry 之间保持契约。

断言可以很简单。重要的是，它指向一个真实的产品预期。

```ts
t.messageIncludes(/蓝|blue|白|方块|square/i);
t.judge.autoevals.closedQA("助手是否始终基于图片内容作答？");
```

这也是为什么 eval authoring 应该尽量接近写单元测试。如果写一个 eval 需要额外平台、脆弱脚本和一整周上下文，它就不会变成日常开发的一部分。

## eval 需要 trace

只有通过率，没有证据，不够。一次运行失败时，我们需要看到它是怎么失败的。

Trace 数据告诉我们最终 verdict 之前发生了什么：消息、工具调用、命令输出、成本、耗时、artifact、diff 和 adapter event。这些细节能把原本混在一起的情况拆开：

- agent 误解了任务。
- 工具返回了坏数据。
- sandbox 根本没启动。
- judge 看错了作用域。
- 断言写得太脆。

这些情况被拆开之后，下一步要做什么就清楚了。

## 目标是更快的反馈回路

eval 的重点不是做一个完美排行榜。重点是缩短一次改动和一个答案之间的距离。

我们能不能升级模型？prompt 有没有改善 workflow？新的 adapter 有没有收集到足够证据？这个 fixture 有没有抓住刚修过的回归？失败来自 agent，还是来自基础设施？

没有 eval，这些问题会滑向手工检查。有了 eval，它们就变成可以留在 repo 里、跑在 CI 里、并且产出可检查 artifact 的重复检查。

这就是为什么我们需要 eval：不是因为 agent 不应该失败，而是因为 agent 工作需要一种方式，能从失败里学习，而不是每次都从零开始。

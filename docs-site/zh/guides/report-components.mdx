报告文件里的每个组件都是双面的：网页面是 React 渲染，终端面是字符渲染，两面吃同一份算好的数据，走哪扇门由宿主决定（见[自定义报告](/zh/guides/custom-reports)）。每个组件的数据算法挂在它自己身上——`<MetricTable data={await MetricTable.data(selection, …)} />`，组件名后打个点就找到，算和画永远配对。本页逐个列出官方组件：它回答什么问题、在报告里怎么调用、终端输出长什么样。

所有组件共守同一套契约，下面不再逐个重复：

- **诚实渲染。** 缺数据渲染 `—` 不补 0；覆盖不全的格子带 `12/15` 角标；截断如实标注剩余数量与原始 artifact 路径。
- **排序与方向随指标的 `better`。** higher 的指标降序、lower 的升序，「好」的一头恒在上、在右上。
- **证据室深链。** 网页面的格子、点和条目深链到 attempt 弹窗；终端面在条目下印 `niceeval show <eval id>` 下钻命令。
- **两面同口径，网页面不依赖 JS 也完整。** 排序在计算时由 `sort` 定死，终端和网页看到同一份基准顺序；下钻是普通链接、展开折叠用 `<details>`。网页面另有一层浏览操作：点表头就地重排、榜单行过滤、图表点位悬停看数值——只影响眼前的视图，不改数据口径，刷新即回基准顺序；浏览器禁用 JS 时这些操作消失，内容一样不少（悬停信息退化为图内提示）。

## 排版原语

`Row` / `Col` / `Section` / `Text` 负责摆版面，一次摆放两个面各自成立：`Col` 纵向堆叠；`Row` 网页横排、终端字符分栏（宽度不够自动降级纵向）；`Section` 是带标题的块；`Text` 是说明文字。另有 `<Style>{css}</Style>` 给自定义组件带样式：web 面吐 `<style>` 标签、text 面渲染为空——静态导出不打包用户代码，`className` 引用的 CSS 靠它随树走。

```tsx
<Col>
  <Section title="考试成绩单">
    <Row>
      <Scoreboard data={board} />
      <Text>algebra 权重 ×2，满分 100。</Text>
    </Row>
  </Section>
</Col>
```

```text
考试成绩单
  agent  总分       algebra  geometry │ algebra 权重 ×2，
  bub    86.5/100   45/50    41.5/50  │ 满分 100。
  codex  78.0/100   40/50    38/50    │
```

## `DefaultReport` —— 官方水位整块

官方水位有两种形式，各管一件事。值 `defaultReport`（从 `niceeval/report` 导出）是官方两扇门裸跑时渲染的默认报告本身——运行总览、每个实验组一节（质量 × 成本散点图 + 榜单，组按 experiment id 的 `/` 前缀划分）、失败清单，形态见[查看结果](/zh/guides/viewing-results)；`niceeval view` 等于 `niceeval view --report` 它。组件 `<DefaultReport />` 是摆进你自己报告里的官方水位锚点块：运行总览、现刻榜单（experiment × evals 折叠计票 / 通过率 / 成本 / 耗时）、失败清单。组件零 props、纯声明——官方水位只读瘦身条目，宿主总是替你备好，经上下文注入；官方口径与你的口径并排对照，正是摆它的理由：

```tsx
<DefaultReport />
```

```text
Current verdicts · 2 experiments · composed from 3 runs · latest 2026-07-09T10-00-00-000Z

experiment              evals   pass   cost    duration
compare/bub-gpt-5.4     13/15   87%    $0.42   4m 12s
compare/codex-gpt-5.4   12/15   80%    $0.51   5m 03s

Failing:
  ✗ weather/brooklyn   compare/codex-gpt-5.4   gate calledTool("get_weather")   41s ago
      → niceeval show weather/brooklyn
```

榜单每行一个 experiment，evals 是按 eval 折叠后的判定计票。要 agent / model 随行列、分组散点这些默认报告的完整形态，用值 `defaultReport`；要在自己口径旁只留官方水位一块，用这个组件。

## `RunOverview` —— 页头 KPI 条

每张报告开头「这批数据是什么」：几个配置、几道题、通过分布、总成本、何时跑的；Selection 里的挑选警告随数据一起进来、直接显示在条内，诚实不靠你记得接线。

```tsx
<RunOverview data={await RunOverview.data(selection)} />
```

```text
2 experiments · 15 evals · 42 attempts · composed from 3 runs · latest 2026-07-09T10-00
passed 26 · failed 12 · errored 4 · skipped 0 · $0.93 · 9m 15s
! snapshot compare/codex-gpt-5.4 predates latest run by 2 days
```

## `GroupSummary` —— 组摘要

一组 experiment（如 `defaultReport` 里同一个 `<Section>` 内的全部 experiment）的紧凑统计块，回答「这组题综合算下来过了多少、花了多少、什么时候跑的」。组件本身不分组——先用 `Selection.filter` 把 Selection 收窄到这一组，再喂给 `GroupSummary.data`：

```tsx
const scoped = selection.filter((s) => s.experimentId.startsWith("compare/"));

<Section title="compare">
  <GroupSummary data={await GroupSummary.data(scoped)} />
</Section>
```

```text
Pass rate 73.3% · 2 experiments · 15 evals · failed 3 · errored 1 · $0.93
latest 2026-07-09T10:00:00Z
```

通过率是 eval 级折叠计票口径：同一 eval 的多轮 attempt 先折成一个判定（任一轮通过则算通过，否则取最严重的），`passed / (passed + failed + errored)`，`skipped` 不进分母——与页头 `RunOverview` 那个两级聚合、带 partial credit 的 `passRate` 是两码事，两者服务不同问题：`GroupSummary` 回答「这组题过了几道」，`RunOverview` 回答「整体质量几分」，不要互相替代。`evals` 按 `experimentId + eval id` 的完整身份键去重——组里两个 experiment 各自的同名 eval 算两道题，不会被误合并成一道。`errored` 为 0 时这一段省略，但 `verdicts.errored` 这个数据字段本身永远在，省略只发生在渲染层。`totalCostUSD` 是组内可测 attempt 成本求和，一个 attempt 都没报成本时是 `null`，两面都渲染缺数据而不是 `$0`。

要在自己的报告文件里摆同一块组摘要（不止 `defaultReport` 能用），直接调 `GroupSummary.data`——它和 `defaultReport` 用的是同一个公开计算函数，不是默认报告独占的能力。

## `ExperimentTable` —— 实验诊断工作台

默认报告的主表不是一张通用指标表，而是完整的实验诊断工作台。一行一个 experiment，关闭时只显示短名称、model、agent、平均耗时、成功率、Tokens、预估成本和一枚判定摘要；点整行展开后，依次看到完整实验配置、汇总 KPI、逐 Eval/Attempt 诊断和原始样例结果。具体证据通过 `AttemptRef` 深链进入查看器的 attempt 详情。

```tsx
<ExperimentTable
  data={await ExperimentTable.data(selection)}
  filter
/>
```

网页面默认按成功率降序，可点击任一表头临时排序；`filter` 加入实验过滤框。无客户端 JavaScript 时仍保留预排后的完整内容，展开使用原生 `<details>`。终端面输出同一份主榜，并在对应 experiment 下列出失败和错误的 Eval。

它与 `MetricTable` 是两种组件：`ExperimentTable` 的展开内容是配置、诊断总量和证据；`MetricTable` 是任意维度 × 指标的通用比较表，`expand` 只会把同一套指标换一个维度重算。需要复刻默认首页时用前者，需要自由换维度和指标时用后者。

## `MetricTable` —— 榜单

一行一个维度值、一列一个指标的对比表，回答「谁整体更好」。行维度、指标列、排序全部可换，自定义指标（`defineMetric`）与内置指标同列。

```tsx
<MetricTable data={await MetricTable.data(selection, {
  rows: "agent",
  columns: [passRate, codeLines, costUSD],
  sort: passRate,
})} />
```

```text
agent   pass rate   code lines   cost
bub     87%         312 lines    $0.42
codex   80% 12/15   355 lines    $0.51
```

两面都按 `sort` 预排，基准顺序一致——要固定换一种排序，改一行重跑。`12/15` 角标表示该格 15 个 attempt 里只有 12 个测得了这个指标。

`rows: "experiment"` 时每行自动带 agent 与 model 列——结果里现成的元信息，不用配置。给组件传 `filter`（`<MetricTable data={…} filter />`）时，网页面在表上方渲染一个过滤框，输入即隐藏不匹配的行——纯浏览操作，不改数据，终端面没有对应物；要固定口径的过滤，用 `evals` 前缀参数或换维度重算。

加一个 `expand`（如 `expand: "eval"`），每行就能点开看更细一级：

```tsx
<MetricTable data={await MetricTable.data(selection, {
  rows: "experiment",
  columns: [passRate, costUSD],
  expand: "eval",
})} filter />
```

点开一行，看到的是这个 experiment 下每道题的判定、失败原因、以及同一套指标列在这道题上单独算出来的值——网页面是原生的展开/收起交互，不用等 JS；终端面是这一行下面缩进的失败清单。子行不是另一套东西，就是同一张表在更细的维度上再算一遍。

## `MetricMatrix` —— 逐题格子

行 × 列两个维度、格子里一个指标，回答「哪道题谁挂了」。稀疏渲染：没有样本的格子空着，不编数。

```tsx
<MetricMatrix data={await MetricMatrix.data(selection, { rows: "eval", columns: "agent", cell: passRate })} />
```

```text
eval                bub    codex
algebra/quadratic   100%   100%
algebra/matrix      100%   67%
geometry/area       50%    —

next: niceeval show geometry/area
```

网页面点格子深链到该格的 attempt；终端面在表下印下钻命令。

## `MetricBars` —— 分组条形

同一份矩阵数据的另一种摆法（`MetricBars.data` 就是 `MetricMatrix.data` 的别名）：按组并排比大小，回答「每个科目上谁领先、差多少」。组维度一组条、系列维度一根条、条长是指标值——benchmark 发布图（Terminal-Bench、BrowseComp 各一组，每个 agent 一根柱）就是这个形状。

```tsx
<MetricBars data={await MetricBars.data(selection, {
  rows: "evalGroup",        // 一组条 = 一个科目/benchmark
  columns: "agent",         // 一根条 = 一个 agent
  cell: passRate,
})} />
```

```text
algebra
  bub     ██████████████████░░  91.9%
  codex   █████████████████░░░  88.0%
geometry
  bub     ██████████░░░░░░░░░░  50.0%
  codex   —
```

网页面是竖向分组柱：柱顶标数值，系列颜色与其它组件的稳定配色一致，图例自动生成。终端面横向条形，字符宽度即刻度，同组内按值排序（方向随 `better`）。`better: "lower"` 的指标（成本、耗时）条形反向填充，短条恒为「好」。

## `Scoreboard` —— 考试成绩单

总分 + 分科小计，回答「这套题它能得几分」。逐题分值制：权重按 eval id 前缀配置，分母对所有被打分者恒定，没跑到的题挣 0 分并如实报 `missing`。

```tsx
<Scoreboard data={await Scoreboard.data(selection, {
  rows: "agent",
  subjects: "evalGroup",
  weights: { "algebra/": 2 },
  fullMarks: 100,
})} />
```

```text
agent   total      algebra        geometry
bub     86.5/100   45/50          41.5/50
codex   71.0/100   40/50          31/50 (1 missing)
```

## `MetricScatter` —— 质量 × 成本 frontier

每个点一个配置、两个指标各占一轴，回答「又好又便宜的是谁」。`series` 把同 agent 不同档位的点连成线；`better` 驱动轴向——`lower` 的轴反向画，「好」的角落恒在右上。

```tsx
<MetricScatter data={await MetricScatter.data(selection, {
  points: "experiment",     // 每个点 = 一个配置的聚合
  series: "agent",          // 同 agent 的档位连成线
  x: costUSD,
  y: passRate,
})} />
```

```text
pass ↑                                    （好 → 右上）
 90%│                                  A
    │                     B
 75%│                          C
 60%│      D
    └──────────────────────────────────────→ cost（轴反向：越右越省）
     $0.60       $0.45       $0.30      $0.15

A bub-high   B bub-medium   C codex-high   D codex-low
```

网页面点带悬停提示（值与 `samples/total`，禁用 JS 时退化为图内提示）、同系列连线、点击深链下钻。终端面用字母标点、图例列在图下；x 或 y 缺数据的点两个面都不画，注脚如实报「n 个点缺数据」；点太密排不下时降级为坐标表，不硬挤。维度槽也收自定义维度（`{ name, of }`，从 attempt 已有数据算组名）和 `flag()`（experiment 声明的变量），怎么选见[自定义报告](/zh/guides/custom-reports)的「换分组」一节。

## `MetricLine` —— 趋势线

x 是有序变量、每个系列一条线，回答「变量拧大，分数怎么走」——并行 agent 数 × 模拟延迟 × 得分这类 scaling 图就是它。与 `MetricScatter` 的分工：scatter 的两轴都是测出来的指标（找 frontier），line 的 x 是你配置的变量（看趋势）。变量在 experiment 的 `flags` 里声明，报告用 `flag()` 直接引用，不从 experiment 命名里解析。

```tsx
<MetricLine data={await MetricLine.data(selection, {
  x: flag("latencyMs", { label: "Simulated latency", unit: "ms" }),
  series: flag("agents", { label: (v) => `${v} agents` }),
  y: passRate,
})} />
```

```text
pass ↑
 80%│                   C···C
    │            C···C            B
 60%│       C         B···B
    │         B···B                        A
 40%│       A···A
    │    A
 20%│
    └───────────────────────────────────────────→ Simulated latency
      1 min       5 min       10 min      20 min

A 1 agents   B 4 agents   C 16 agents
```

每个点是一个 experiment 的聚合，与其它组件同一台两级聚合引擎，`samples` / `total` / `refs` 一应俱全；同系列的点按 x 排序连线。网页面每个点可 hover、可深链下钻。终端面同系列共用一个字母、沿 x 排布，趋势肉眼可读；y 缺数据的点不画、注脚报数，点太密降级为坐标表。

## `DeltaTable` —— 成对对比

每行一对配置、每列一个指标，格子里 A、B、Δ 三个值，回答「这个开关值不值」「这次修复翻转了什么」。涨跌好坏由 `better` 判定，任一侧缺数据 Δ 显示为缺，不硬算。「这次 vs 上次」（同一配置的两个结果快照）也是它：`pairs` 的 `a` / `b` 除 experiment id 外也收快照键 `<experimentId> @ <startedAt>`——时间轴对比要用到旧快照，`results.latest()` 里没有，配手挑的快照数组（如 `[exp.latest, exp.snapshots[1]]`）用。

```tsx
<DeltaTable data={await DeltaTable.data(selection, {
  pairs: [
    { a: "compare/bub", b: "compare/bub--agents-md", label: "bub" },
    { a: "compare/codex", b: "compare/codex--agents-md", label: "codex" },
  ],
  metrics: [passRate, costUSD],
})} />
```

```text
pair    pass rate              cost
bub     87% → 93%   +6%        $0.42 → $0.45   +$0.03
codex   80% → 80%   ±0         $0.51 → —       —
```

## `CaseList` —— 失败案例清单

失败与出错的 attempt 逐条列出：失败断言、error 摘要、judge 评语。榜单回答「多少」，它回答「为什么」。

```tsx
<CaseList data={await CaseList.data(selection, { verdicts: ["failed", "errored"], limit: 20 })} />
```

```text
✗ weather/brooklyn · compare/codex-gpt-5.4 · failed · 41s · $0.04
    gate calledTool("get_weather") — tool was never called
    soft judge("回答基于实时数据") — 0.2/1: reply invents a temperature
    → niceeval show weather/brooklyn
✗ fixtures/button · compare/bub-gpt-5.4 · failed · 2m 10s · $0.09
    gate fileChanged("src/components/Button.tsx")
    → niceeval show fixtures/button

(3 more not shown · limit 20)
```

自由文本（error、断言 detail、judge 评语）发布前要消毒时，用 `CaseList.data` 的 `redact` 钩子。

## 官方组件之外

以上摆法都表达不了时，用 `defineComponent` 写自己的双面组件——网页怎么渲染、终端字符怎么排，两个面你都说了算，写法见[自定义报告](/zh/guides/custom-reports)的「换形态」一节。

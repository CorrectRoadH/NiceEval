对比对象：

- **before**：[`examples/zh/origin/pi-sdk/`](https://github.com/CorrectRoadH/niceeval/tree/main/examples/zh/origin/pi-sdk) —— 一个独立的 `@earendil-works/pi-agent-core` HTTP 服务（`src/backend/server.ts`/`agent.ts`/`tools.ts`），还没接任何 eval。
- **after**：[`examples/zh/tier1/pi-sdk/`](https://github.com/CorrectRoadH/niceeval/tree/main/examples/zh/tier1/pi-sdk) —— 同一个应用接入 NiceEval 之后的样子。

应用侧 `src/backend/*` **逐字节未变**——`agents/server-lifecycle.ts` 按需拉起它的
`server.ts` 子进程、轮询 `/healthz`；`agents/pi-sdk.ts` 手写把原生 `AgentEvent`
SSE 流（`message_end` / `tool_execution_start` / `tool_execution_end`，外加
`session` / `approval_request` / `server_error` 三种传输层帧）翻成标准事件流。
这个应用没有 OTel（pi-agent-core 没有官方集成），事件断言全靠这层手工映射；
`calculate` 工具挂了 HITL 审批，approve/deny 靠模块级 Map 存着一条还开着的 SSE
流，续读不重发。

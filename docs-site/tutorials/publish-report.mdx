`niceeval view --out <dir>` exports the viewer into a plain static directory: reports, snapshot lists, transcripts, and the trace waterfall, exactly like what local `niceeval view` shows (export behavior is covered in [Viewing Results · Exporting and Static Hosting](/guides/viewing-results#exporting-and-static-hosting)). Publishing through CI only requires getting the result data into the build machine's hands — but do not commit the local source of truth `.niceeval/` directly: per-string truncation can stop one runaway output from ballooning, but it cannot guarantee a whole file stays under a Git host's limit. First use `copySnapshots` to produce a publish results root that has passed the 50 MiB per-file precheck, then commit that directory.

## Producing a committable results directory

Create `scripts/publish-results.mjs`:

```javascript
import { rm } from "node:fs/promises";
import { copySnapshots, openResults } from "niceeval/results";

const output = "report-data";
const results = await openResults(".niceeval");

await rm(output, { recursive: true, force: true });
await copySnapshots(results.latest(), output, {
  artifacts: ["sources", "events", "trace", "o11y", "agentSetup"],
});
```

Run `node scripts/publish-results.mjs`, then commit `report-data/`. `copySnapshots` checks every file it is about to publish before creating the directory; if any file exceeds 50 MiB, the whole copy fails, lists the path, the size, and what to do about it, and leaves no half-written directory behind. `diff` is not published by default; add it to `artifacts` explicitly when you need it, and it is bound by the same budget. Oversized events / trace left behind by an older version are never silently rewritten — the precheck asks you to exclude that class of evidence or rerun with the current version.

## The build command is the export command

```bash
npx niceeval view --run report-data --out site
```

`view` fails outright with a non-zero exit code on zero readable results — it never exports an empty report. If the `report-data/` checkout is broken, or every stored snapshot is incompatible with the current niceeval's `schemaVersion` and gets skipped wholesale, the build fails and Vercel / GitHub Pages keeps the previous deployment. The error lists every skipped snapshot directory with its reason, and for the `schemaVersion` case also gives the `npx niceeval@<version> view` command that can view the old data directly.

## Publishing a custom report

Without `--report`, the published site is the default report's three pages (Report, Attempts, Traces). To switch to your own pages, pass a [`defineReport` report file](/guides/custom-reports) to `--report` — attempt detail (transcript, trace, code view) stays in the same site, and every number in the report drills into its corresponding evidence, exactly like local `view --report`:

```bash
npx niceeval view --run report-data --report reports/exam.tsx --out site
```

The report file is committed in the repo just like `report-data/` — edit the layout, push, and the live site updates. With the `vercel.json` / workflow below, just swap the build command for this one line; nothing else needs to change.

## Connecting a hosting platform

**Vercel**: drop a `vercel.json` at the repo root; after importing the project, push to deploy.

```json
{
  "installCommand": "pnpm install --frozen-lockfile",
  "buildCommand": "npx niceeval view --run report-data --out site",
  "outputDirectory": "site"
}
```

**GitHub Pages**: in the repo's Settings → Pages, set Source to GitHub Actions, then add a workflow:

```yaml
# .github/workflows/report.yml
name: report
on:
  push:
    branches: [main]

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - run: npx niceeval view --run report-data --out site
      - uses: actions/upload-pages-artifact@v3
        with:
          path: site
  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - id: deployment
        uses: actions/deploy-pages@v4
```

The daily loop is: run the eval locally, run `node scripts/publish-results.mjs`, commit the updated `report-data/`, and push. The publish directory keeps only the latest snapshot per experiment; to publish under a different selection policy, swap out `results.latest()` in the script. Your local `.niceeval/` can keep the full history and diffs — there is no need to cut away debugging evidence just to satisfy a Git limit.

## What gets published is the selected evidence

The full site export carries the transcripts, source snapshots, and traces selected in the publish results root. Newly written oversized events / trace strings may carry structured truncation markers; `copySnapshots` does not rewrite content — it only selects and runs the whole-file size precheck. Before publishing to the public internet, confirm the results contain no secrets or sensitive data; for redacting free text on custom report pages, see the `redact` hook in [Results Data API](/guides/results-data).
